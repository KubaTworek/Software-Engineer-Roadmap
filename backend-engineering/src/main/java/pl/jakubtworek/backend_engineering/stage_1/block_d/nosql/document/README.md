# Document databases

Document database to typ bazy NoSQL, w którym podstawową jednostką zapisu i odczytu jest dokument. Najczęściej taki dokument przypomina strukturę JSON albo BSON. Przykładami baz dokumentowych są MongoDB, Couchbase i Firestore. Ten model jest szczególnie wygodny wtedy, gdy dane mają naturalnie zagnieżdżoną strukturę, nie są idealnie tabelaryczne albo często są odczytywane jako większy agregat.

Najważniejsza różnica względem SQL polega na tym, że w bazie dokumentowej bardzo często projektujesz dokument pod sposób użycia danych. W SQL naturalne jest rozdzielanie danych na tabele, normalizacja i łączenie ich za pomocą joinów. W document DB częściej pytasz: „jakie dane muszę pobrać razem?”. Jeżeli endpoint szczegółów zamówienia zawsze potrzebuje zamówienia, pozycji zamówienia, nazwy produktu i historycznego emaila użytkownika, to sensowne może być trzymanie części tych danych razem w jednym dokumencie zamówienia.

Embedowanie danych oznacza umieszczanie jednych struktur wewnątrz drugich. Przykładowo dokument zamówienia może zawierać tablicę pozycji `items`. To jest naturalny model, jeśli pozycje zamówienia nie istnieją samodzielnie z perspektywy większości operacji i zawsze są czytane razem z zamówieniem. Dzięki temu można pobrać pełne zamówienie jednym zapytaniem bez joinów i bez wielu round-tripów do bazy. To jest jedna z największych zalet document DB.

Denormalizacja jest w tym świecie normalnym narzędziem. Dokument zamówienia może przechowywać `userEmail`, `productName` i `unitPrice`, mimo że te dane istnieją też w innych miejscach. W SQL taka duplikacja często byłaby traktowana podejrzanie. W document DB może być świadomą decyzją, szczególnie gdy dokument ma reprezentować historyczny snapshot. Jeśli użytkownik zmieni email po złożeniu zamówienia, stare zamówienie nadal może zawierać email użyty w momencie zakupu. To nie musi być niespójność, tylko poprawny zapis faktu historycznego.

Nie oznacza to jednak, że wszystko należy embedować. Zbyt duże dokumenty są problemem. Jeżeli dokument użytkownika zawierałby całą historię zamówień, eventów, logowań, powiadomień i rekomendacji, mógłby rosnąć bez kontroli. Duży dokument jest wolniejszy do przesłania, trudniejszy do aktualizacji i bardziej podatny na konflikty zapisu. Embedowanie ma sens wtedy, gdy dane są ograniczone rozmiarem, często czytane razem i aktualizowane w podobnym cyklu życia.

Referencje nadal są potrzebne. Jeżeli dana encja jest duża, współdzielona przez wiele dokumentów, aktualizowana niezależnie albo często wyszukiwana samodzielnie, lepiej trzymać ją osobno i przechowywać tylko identyfikator. Przykładowo produkt w katalogu może być osobnym dokumentem, a zamówienie przechowuje tylko wybrane historyczne dane produktu. Decyzja między embedowaniem a referencją zależy od access patternów, rozmiaru danych i wymagań spójności.

Indeksy w document DB są równie ważne jak w SQL, ale trzeba rozumieć je w kontekście dokumentów. W MongoDB można tworzyć indeksy pojedyncze, compound indexes i multikey indexes dla tablic. Kolejność pól w compound index ma znaczenie. Indeks `{ userId: 1, status: 1, createdAt: -1 }` dobrze pasuje do zapytań filtrujących po `userId`, potem po `status`, i sortujących po `createdAt`. Nie musi jednak dobrze wspierać zapytań tylko po `status`, bo pomijają pierwsze pole indeksu. Sam fakt istnienia indeksu nie gwarantuje poprawy wydajności; trzeba sprawdzić plan przez `explain()`.

Document DB dobrze pasuje do systemów, w których dane są agregatami: zamówienia, profile, katalogi produktów, konfiguracje, dokumenty użytkownika, treści CMS, formularze, zdarzenia domenowe albo dane półstrukturalne. Dobrze sprawdza się też tam, gdzie schemat ewoluuje i nie wszystkie rekordy mają identyczny zestaw pól. To nie znaczy, że schemat nie istnieje. Schemat istnieje, tylko często jest egzekwowany bardziej po stronie aplikacji niż przez bazę.

Typowym błędem jest traktowanie MongoDB jak relacyjnej bazy bez joinów. Jeśli tworzymy wiele małych kolekcji i potem ręcznie łączymy dane w aplikacji przez serię zapytań, łatwo wpaść w NoSQL-owy odpowiednik problemu N+1. Innym błędem jest bezrefleksyjne embedowanie wszystkiego w jeden dokument. Oba podejścia ignorują sedno document DB, czyli projektowanie dokumentu wokół konkretnych odczytów i zapisów.

Najważniejsze pytania projektowe brzmią: czy te dane są czytane razem, czy aktualizują się razem, czy dokument ma naturalną granicę, jak duży może urosnąć, czy potrzebuję query po zagnieżdżonych polach, jakie indeksy będą potrzebne i czy akceptuję duplikację. Dobra baza dokumentowa nie jest „SQL-em w JSON-ie”. To inny model pracy, który może być bardzo wydajny, jeśli struktura dokumentów odpowiada realnym access patternom aplikacji.
