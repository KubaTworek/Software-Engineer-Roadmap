# Modelowanie danych w NoSQL

Modelowanie danych w NoSQL różni się od klasycznego modelowania relacyjnego. W SQL naturalnym punktem startu jest model domenowy, encje, relacje, normalizacja i integralność danych. W NoSQL bardzo często punktem startu są access patterny, czyli konkretne sposoby, w jakie aplikacja będzie czytać i zapisywać dane. To jest fundamentalna zmiana myślenia. Zamiast pytać „jak wygląda idealny model danych?”, pytasz „jakie zapytania system musi obsłużyć szybko i niezawodnie?”.

Access pattern opisuje konkretną operację na danych. Przykładem może być: pobierz zamówienia użytkownika posortowane od najnowszych, pobierz szczegóły zamówienia po ID, zapisz event użytkownika, pobierz metryki urządzenia z ostatniej godziny, znajdź aktywną sesję po tokenie albo pobierz rekomendacje dla produktu. Każdy z tych wzorców może prowadzić do innego modelu danych. W NoSQL jedna struktura danych często nie służy wszystkim możliwym zapytaniom. Czasem tworzy się osobne read modele pod konkretne widoki.

Denormalizacja jest naturalną konsekwencją projektowania pod odczyt. Jeśli endpoint potrzebuje danych zamówienia, nazwy produktu i emaila użytkownika, można te informacje skopiować do dokumentu zamówienia albo read modelu. Dzięki temu odczyt jest prosty i szybki. Kosztem jest duplikacja oraz ryzyko niespójności. Jeżeli email użytkownika zmieni się w profilu, kopie w innych miejscach nie zaktualizują się magicznie. Trzeba zdecydować, czy to problem. W przypadku historycznego zamówienia stary email może być poprawny. W przypadku aktualnego profilu publicznego może już nie być.

Embedowanie danych oznacza umieszczanie jednych danych wewnątrz innych, na przykład pozycji zamówienia wewnątrz dokumentu zamówienia. To dobrze działa, gdy dane są czytane razem, mają wspólny cykl życia i nie rosną bez limitu. Referencje są lepsze, gdy dane są duże, współdzielone, aktualizowane niezależnie albo często wyszukiwane samodzielnie. Nie ma jednej uniwersalnej reguły. Każda decyzja powinna wynikać z access patternów i wymagań spójności.

W modelowaniu NoSQL bardzo ważne są granice agregatu. Agregat to grupa danych, którą traktujemy jako logiczną całość. Zamówienie z pozycjami może być agregatem. Profil użytkownika z preferencjami też może nim być. Cała historia aktywności użytkownika raczej nie powinna być jednym agregatem, bo może rosnąć bez końca. Dobre wyznaczenie granic agregatu pomaga uniknąć zbyt dużych dokumentów, hot partitions i konfliktów zapisu.

Partition key albo shard key to kolejny krytyczny element modelu. Decyduje, jak dane zostaną rozłożone między partycje, shardy lub nody. Dobry klucz ma wysoką kardynalność, równomiernie rozkłada ruch i pasuje do zapytań. Zły klucz może stworzyć hot partition, czyli sytuację, w której większość ruchu trafia do jednego miejsca. Przykładowo `status` jako partition key jest zwykle słabym wyborem, bo wartości jest niewiele, a większość rekordów może mieć status `ACTIVE`. Lepszym kandydatem może być `userId`, `tenantId`, `deviceId` albo połączenie identyfikatora z bucketem czasowym, ale to zależy od danych.

NoSQL często wymaga myślenia o write amplification i read amplification. Write amplification pojawia się wtedy, gdy jeden zapis logiczny trzeba zapisać w kilku miejscach, na przykład w głównym dokumencie i kilku read modelach. Read amplification pojawia się wtedy, gdy jeden odczyt biznesowy wymaga wielu zapytań do bazy. SQL często zmniejsza duplikację i pozwala łączyć dane joinami, ale może mieć kosztowne zapytania. NoSQL często przyspiesza odczyt przez duplikację, ale komplikuje zapis.

Event-driven architecture, outbox pattern i projekcje są często używane razem z NoSQL. Główna operacja zapisuje fakt biznesowy, a następnie event aktualizuje modele odczytowe. Taki system może być eventual consistent, więc read model przez chwilę pokazuje starszy stan. To jest akceptowalne tylko wtedy, gdy domena na to pozwala. Trzeba projektować idempotencję, retry i obsługę opóźnionych eventów. Bez tego denormalizacja szybko zmienia się w chaos.

Nie należy wybierać NoSQL tylko dlatego, że „SQL się nie skaluje”. Relacyjne bazy danych potrafią skalować się bardzo dobrze i często są prostszym wyborem. NoSQL ma sens wtedy, gdy konkretny model bazy pasuje do problemu: dokumenty do agregatów, key-value do lookupów po kluczu, wide-column do ogromnych strumieni zapisu i time-series, graph do relacji i traversal queries. Zły wybór NoSQL może być trudniejszy w utrzymaniu niż dobrze zaprojektowany SQL.

Praktyczne modelowanie powinno zaczynać się od listy access patternów. Dla każdego patternu warto opisać typ operacji, klucz dostępu, sortowanie, wolumen danych, wymaganą spójność, oczekiwane latency i częstotliwość. Dopiero potem dobiera się typ bazy, strukturę kluczy, indeksy, denormalizację i ewentualne read modele. Jeśli nie znasz access patternów, projekt NoSQL będzie zgadywaniem.

Najważniejszy wniosek jest taki, że NoSQL przesuwa ciężar decyzji z bazy na projekt aplikacji. Dostajesz większą elastyczność i potencjalnie lepszą skalowalność, ale płacisz za to koniecznością świadomego modelowania, obsługi niespójności i projektowania danych pod konkretne zapytania. Dobre NoSQL nie polega na braku schematu. Polega na tym, że schemat jest podporządkowany sposobowi użycia danych.
