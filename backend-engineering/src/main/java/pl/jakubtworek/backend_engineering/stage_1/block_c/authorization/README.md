# Bezpieczeństwo w Spring – JWT, OAuth2 i @PreAuthorize

Nowoczesne aplikacje backendowe bardzo często wykorzystują architekturę stateless, w której serwer nie przechowuje sesji użytkownika. Zamiast klasycznej sesji HTTP używa się tokenów JWT przekazywanych w każdym żądaniu. Spring Security wraz z OAuth2 dostarcza kompletny mechanizm uwierzytelniania i autoryzacji oparty o tokeny, role oraz uprawnienia. :contentReference[oaicite:0]{index=0}

JWT, czyli JSON Web Token, jest podpisanym tokenem zawierającym informacje o użytkowniku oraz jego uprawnieniach. Token składa się z trzech części: nagłówka, payloadu oraz podpisu. W payloadzie znajdują się tzw. claims, czyli dane takie jak login użytkownika, role, uprawnienia czy data wygaśnięcia tokena. Token jest podpisywany kluczem prywatnym serwera, dzięki czemu Resource Server może później zweryfikować jego autentyczność przy użyciu klucza publicznego. :contentReference[oaicite:1]{index=1}

W praktyce stosuje się dwa rodzaje tokenów: Access Token oraz Refresh Token. Access Token jest krótkotrwały i służy do autoryzacji żądań HTTP. Najczęściej jego czas życia wynosi od kilku do kilkunastu minut. Dzięki temu nawet jeśli token zostanie przechwycony, jego użyteczność jest ograniczona czasowo. Refresh Token ma znacznie dłuższy czas życia i służy wyłącznie do uzyskania nowego Access Tokena bez konieczności ponownego logowania użytkownika. :contentReference[oaicite:2]{index=2}

Bardzo ważnym mechanizmem bezpieczeństwa jest refresh token rotation. Polega on na tym, że przy każdym użyciu Refresh Tokena serwer generuje nowy Refresh Token i jednocześnie unieważnia poprzedni. Dzięki temu przechwycony wcześniej token staje się bezużyteczny po pierwszym użyciu. W praktyce oznacza to konieczność przechowywania Refresh Tokenów po stronie serwera, najczęściej w bazie danych. Dodatkowo w bezpiecznych implementacjach nie zapisuje się pełnych tokenów, lecz ich hashe. :contentReference[oaicite:3]{index=3}

W Spring Security JWT najczęściej działa w modelu OAuth2 Resource Server. W tym podejściu występują dwa główne komponenty: Authorization Server oraz Resource Server. Authorization Server odpowiada za uwierzytelnienie użytkownika i wydawanie tokenów. Resource Server jest aplikacją backendową, która przyjmuje token JWT w nagłówku `Authorization: Bearer ...` i weryfikuje jego poprawność. Spring Boot potrafi skonfigurować Resource Server praktycznie automatycznie po podaniu `issuer-uri`. Framework pobiera wtedy konfigurację OpenID Connect oraz publiczne klucze JWKS służące do weryfikacji podpisów tokenów. :contentReference[oaicite:4]{index=4}

JWT jest rozwiązaniem stateless, co oznacza, że serwer nie przechowuje informacji o aktywnych sesjach użytkowników. Każde żądanie zawiera komplet danych potrzebnych do autoryzacji. Z jednej strony zwiększa to skalowalność systemu i dobrze współpracuje z architekturą mikroserwisową, ale z drugiej utrudnia natychmiastowe unieważnianie tokenów. Jeśli istnieje potrzeba natychmiastowego wylogowania użytkownika, można zastosować blacklistę tokenów lub skrócić czas życia Access Tokenów. :contentReference[oaicite:5]{index=5}

Po poprawnej walidacji JWT Spring Security tworzy obiekt `Authentication` i umieszcza go w `SecurityContext`. Od tego momentu aplikacja może wykonywać autoryzację na podstawie ról i uprawnień użytkownika. Najczęściej wykorzystywane są do tego adnotacje `@PreAuthorize`. :contentReference[oaicite:6]{index=6}

`@PreAuthorize` pozwala definiować reguły dostępu bezpośrednio na poziomie metod serwisowych lub kontrolerów. Mechanizm ten wykorzystuje Spring Expression Language, czyli SpEL. Możliwe jest sprawdzanie ról, uprawnień, danych użytkownika lub nawet wykonywanie własnej logiki biznesowej. Wyrażenia takie jak `hasRole('ADMIN')` sprawdzają obecność roli użytkownika, natomiast `hasAuthority('ORDER_READ')` pozwala operować bardziej granularnymi uprawnieniami. :contentReference[oaicite:7]{index=7}

W bardziej zaawansowanych scenariuszach autoryzacja może opierać się na danych biznesowych. Przykładowo użytkownik może edytować wyłącznie własne zamówienia. W takich przypadkach Spring pozwala odwoływać się w SpEL do metod własnych beanów, np. `@userSecurity.isOwner(authentication, #orderId)`. Dzięki temu logika autoryzacji zostaje wydzielona do osobnego komponentu i pozostaje łatwa do testowania oraz utrzymania. :contentReference[oaicite:8]{index=8}

Bardzo istotne jest poprawne mapowanie claims z JWT na Spring Security authorities. Spring Security oczekuje obiektów `GrantedAuthority`, dlatego role i uprawnienia zawarte w tokenie muszą zostać odpowiednio skonwertowane. Dla `hasRole('ADMIN')` Spring oczekuje authority `ROLE_ADMIN`, natomiast `hasAuthority('ORDER_READ')` wymaga dokładnie takiej wartości authority. :contentReference[oaicite:9]{index=9}

Jedną z najczęstszych pułapek związanych z `@PreAuthorize` jest brak włączenia method-level security. W Spring Security 6 należy użyć `@EnableMethodSecurity`. Bez tej konfiguracji adnotacje `@PreAuthorize` będą całkowicie ignorowane. Częstym problemem są również błędy w wyrażeniach SpEL, niepoprawne nazwy beanów lub mylenie `authentication` z `principal`. :contentReference[oaicite:10]{index=10}

Warto również pamiętać, że Spring Security działa w oparciu o proxy oraz AOP. Oznacza to, że podobnie jak w przypadku `@Transactional`, autoryzacja metod nie będzie działała poprawnie przy self invocation, czyli wewnętrznym wywołaniu metod tej samej klasy. :contentReference[oaicite:11]{index=11}

Architektura oparta o JWT i OAuth2 jest obecnie standardem w nowoczesnych systemach rozproszonych i mikroserwisowych. Zapewnia wysoką skalowalność, separację odpowiedzialności pomiędzy Authorization Server i Resource Server oraz możliwość centralnego zarządzania uwierzytelnianiem użytkowników. Jednak poprawne wdrożenie wymaga zrozumienia mechanizmów proxy Springa, zasad działania tokenów, rotacji Refresh Tokenów oraz sposobu mapowania ról i uprawnień na mechanizmy autoryzacji Spring Security. :contentReference[oaicite:12]{index=12}
