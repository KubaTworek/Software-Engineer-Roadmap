# Graph databases

Graph database to baza danych zaprojektowana wokół relacji. W SQL relacje też istnieją, ale zwykle są reprezentowane przez klucze obce i joiny między tabelami. W graph DB relacje są obiektami pierwszej klasy. Dane składają się z node’ów, czyli węzłów, oraz edge’y, czyli krawędzi łączących węzły. Przykładami baz grafowych są Neo4j, JanusGraph i Amazon Neptune.

Grafowy model danych jest naturalny wtedy, gdy najważniejsze pytania dotyczą powiązań. Użytkownik może obserwować innych użytkowników, kupować produkty, używać urządzeń, logować się z adresów IP, należeć do organizacji, posiadać role, wchodzić w interakcje z treściami albo być powiązany z innymi kontami. W klasycznym SQL da się to modelować tabelami pośrednimi, ale zapytania przechodzące przez wiele poziomów relacji mogą stawać się skomplikowane i kosztowne. W graph DB takie przejścia są podstawowym sposobem pracy.

Najprostszy przykład to social graph. Mamy relację `(User)-[:FOLLOWS]->(User)`. Pytanie „znajdź znajomych znajomych” oznacza przejście przez dwie relacje. Pytanie „znajdź osoby, które obserwują podobne konta” oznacza analizę wspólnych sąsiadów w grafie. To jest typ problemu, w którym graf jest bardziej naturalny niż zestaw joinów na tabelach relacyjnych.

Drugim częstym use case’em są rekomendacje. Można modelować relacje `(User)-[:BOUGHT]->(Product)`, `(User)-[:VIEWED]->(Product)` i `(Product)-[:IN_CATEGORY]->(Category)`. Na tej podstawie da się szukać produktów kupowanych przez podobnych użytkowników, produktów podobnych do tych już obejrzanych albo kategorii powiązanych z aktywnością użytkownika. Oczywiście rekomendacje można robić też innymi technikami, ale graph DB jest dobrym narzędziem do eksplorowania relacji i szybkiego prototypowania takich zapytań.

Bardzo ważnym zastosowaniem jest fraud detection. Jeśli wiele kont używa tego samego urządzenia, tego samego adresu IP, tej samej karty płatniczej albo tego samego adresu dostawy, graf bardzo szybko pokazuje podejrzane klastry. Relacje typu `(User)-[:USES]->(Device)`, `(User)-[:PAID_WITH]->(Card)` i `(User)-[:LOGGED_IN_FROM]->(IpAddress)` pozwalają wykrywać wzorce, które w tabelach byłyby trudniejsze do zauważenia. W fraud detection często nie chodzi o pojedynczy rekord, ale o strukturę powiązań.

Graph DB dobrze sprawdza się też w dependency graph. Można modelować zależności między serwisami, bibliotekami, modułami, jobami, pipeline’ami albo zasobami infrastruktury. Pytanie „co się zepsuje, jeśli usuniemy ten komponent?” jest naturalnym traversal query. W SQL też da się je wykonać, szczególnie przy rekursywnych CTE, ale graf często daje bardziej czytelny model i wygodniejszy język zapytań.

Nie należy jednak wybierać graph DB do wszystkiego. Jeśli aplikacja wykonuje głównie prosty CRUD po ID, klasyczne transakcje, raportowanie tabelaryczne albo operacje na agregatach bez głębokich relacji, baza grafowa może być przerostem formy nad treścią. Graph DB nie jest domyślnym zamiennikiem SQL. Jest wyspecjalizowanym narzędziem do problemów, w których relacje i przechodzenie po nich są centralnym wymaganiem.

W modelowaniu grafowym trzeba uważać na zbyt ogólne relacje. Relacja powinna coś znaczyć biznesowo. `CONNECTED_TO` jest zwykle mniej użyteczne niż `BOUGHT`, `FOLLOWS`, `DEPENDS_ON`, `USES` albo `BELONGS_TO`. Nazwa relacji powinna pomagać zrozumieć, jaki typ połączenia istnieje między węzłami i jak można go użyć w zapytaniach. Podobnie node’y powinny reprezentować istotne pojęcia domenowe, a nie przypadkowe struktury techniczne.

Wydajność graph DB zależy od tego, jak głęboko i szeroko przechodzimy po grafie. Traversal przez kilka relacji może być bardzo szybki, ale zapytanie, które eksploruje ogromną część grafu bez ograniczeń, może być kosztowne. Tak samo jak w SQL trzeba rozumieć plan zapytania, selektywność startowego node’a i ograniczenia ścieżki. Dobre zapytanie grafowe zwykle zaczyna od możliwie konkretnego punktu startowego i dopiero potem przechodzi po relacjach.

Najważniejsze pytania przed wyborem graph DB są proste: czy problem jest naprawdę relacyjny, czy query często przechodzi przez wiele poziomów powiązań, czy potrzebujemy analizować ścieżki, sąsiedztwa lub klastry, czy relacje mają własne właściwości, i czy SQL joiny zaczynają być głównym źródłem złożoności. Jeśli odpowiedź brzmi tak, graph DB może być bardzo dobrym wyborem. Jeśli nie, prostsza baza relacyjna albo dokumentowa prawdopodobnie będzie łatwiejsza w utrzymaniu.
