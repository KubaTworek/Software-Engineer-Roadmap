# SQL — bazy relacyjne i performance

SQL-owe bazy danych są naturalnym wyborem wtedy, gdy aplikacja pracuje na danych o wyraźnej strukturze, stabilnych relacjach i istotnych wymaganiach dotyczących spójności. Relacyjny model danych dobrze pasuje do systemów, w których encje są ze sobą powiązane, a poprawność tych powiązań jest ważna biznesowo. Przykładami są zamówienia, płatności, faktury, rezerwacje, konta użytkowników, uprawnienia, systemy CRM, ERP albo moduły billingowe. W takich systemach baza danych nie jest tylko miejscem przechowywania rekordów, ale również mechanizmem pilnującym integralności danych.

Największą siłą SQL jest możliwość modelowania danych przez tabele, relacje, klucze główne, klucze obce, ograniczenia unikalności i transakcje. Dzięki temu wiele reguł spójności można przenieść bliżej danych, zamiast polegać wyłącznie na logice aplikacji. Jeżeli zamówienie musi wskazywać istniejącego użytkownika, płatność musi być przypisana do konkretnego zamówienia, a email użytkownika ma być unikalny, relacyjna baza danych daje do tego gotowe mechanizmy. To nie znaczy, że SQL automatycznie rozwiązuje wszystkie problemy projektowe, ale daje solidne fundamenty do budowania systemów, w których dane muszą pozostać poprawne nawet przy równoczesnym dostępie wielu użytkowników.

Performance w SQL najczęściej zaczyna się od zrozumienia zapytań i indeksów. Indeks B-tree pomaga wtedy, gdy baza może dzięki niemu ograniczyć liczbę przeglądanych rekordów, szybciej znaleźć dane pasujące do warunku, efektywniej wykonać join albo uniknąć kosztownego sortowania. Indeks nie jest jednak darmowy. Każdy dodatkowy indeks przyspiesza pewne odczyty, ale spowalnia zapisy, ponieważ przy `INSERT`, `UPDATE` i `DELETE` baza musi aktualizować nie tylko tabelę, ale również struktury indeksowe. Dlatego indeksów nie dodaje się „na wszelki wypadek”. Dobry indeks jest odpowiedzią na konkretne zapytanie, konkretny filtr, konkretny join albo konkretny problem widoczny w planie wykonania.

Szczególnie ważne są indeksy złożone, czyli composite indexes. W ich przypadku kolejność kolumn ma duże znaczenie. Indeks zbudowany jako `(user_id, status, created_at)` nie jest tym samym co indeks `(created_at, status, user_id)`. Baza może efektywnie używać indeksu tylko wtedy, gdy zapytanie pasuje do jego struktury. W praktyce trzeba rozumieć zasadę lewego prefiksu, selektywność kolumn i wpływ warunków równości oraz zakresów na użycie indeksu. Często kolumny używane w warunkach równości powinny znaleźć się przed kolumnami używanymi do zakresów lub sortowania, ale nie jest to mechaniczna reguła. Ostateczna decyzja powinna wynikać z rzeczywistych zapytań i planów wykonania.

Do analizy wydajności zapytań podstawowym narzędziem jest `EXPLAIN ANALYZE`. Samo `EXPLAIN` pokazuje przewidywany plan, natomiast `EXPLAIN ANALYZE` wykonuje zapytanie i pokazuje rzeczywisty czas oraz liczbę przetworzonych rekordów. Dzięki temu można porównać estymacje planera z rzeczywistością. Jeżeli baza zakładała, że przetworzy kilkadziesiąt wierszy, a faktycznie przetwarza miliony, problemem może być brak statystyk, zły indeks, zła selektywność albo zapytanie napisane w sposób utrudniający optymalizację. W planie warto umieć rozpoznać sequential scan, index scan, nested loop, hash join, merge join, sortowanie, filtrowanie po odczycie i miejsca, w których liczba przetwarzanych rekordów jest dużo większa niż liczba rekordów zwróconych.

Sequential scan nie zawsze jest błędem. Jeżeli tabela jest mała albo zapytanie zwraca dużą część danych, pełne skanowanie tabeli może być tańsze niż użycie indeksu. Błędem jest traktowanie każdego sequential scan jako problemu i każdego index scan jako sukcesu. Index scan też może być wolny, jeśli prowadzi do wielu losowych odczytów albo jeśli zapytanie zwraca dużą część tabeli. Dobry backend engineer nie zgaduje, tylko patrzy na plan, czas wykonania, liczbę przetworzonych rekordów i kontekst biznesowy zapytania.

Drugim dużym obszarem SQL są transakcje. Transakcja pozwala wykonać kilka operacji jako jedną logiczną całość. Jeżeli jedna część operacji się nie powiedzie, całość może zostać wycofana. To jest kluczowe przy płatnościach, aktualizacjach salda, rezerwacjach, zmianach stanów magazynowych i wszędzie tam, gdzie częściowy zapis mógłby zepsuć dane. Samo użycie transakcji nie wystarczy jednak do pełnego bezpieczeństwa. Trzeba jeszcze rozumieć poziomy izolacji oraz to, jakie anomalie mogą wystąpić przy równoczesnych operacjach.

`READ_COMMITTED` jest częstym domyślnym poziomem izolacji. Oznacza, że zapytanie widzi tylko dane zatwierdzone przed rozpoczęciem danej instrukcji. Daje to dobry throughput i zwykle wystarcza dla wielu standardowych operacji aplikacyjnych, ale może dopuszczać sytuacje, w których dwa odczyty w tej samej transakcji zobaczą różne dane. `REPEATABLE_READ` daje stabilniejszy obraz danych w ramach transakcji, ale może zwiększać koszt współbieżności i ryzyko konfliktów. Dobór poziomu izolacji nie powinien wynikać z przyzwyczajenia, tylko z konkretnego problemu. Innych gwarancji wymaga generowanie raportu, innych zakup ostatniej sztuki produktu, a jeszcze innych zwykła aktualizacja profilu użytkownika.

Locking jest praktycznym uzupełnieniem transakcji. Optimistic locking sprawdza się, gdy konflikty są rzadkie. Zwykle opiera się na polu `version`, które jest sprawdzane przy zapisie. Jeżeli ktoś wcześniej zmienił rekord, zapis się nie powiedzie i aplikacja musi obsłużyć konflikt, najczęściej przez retry albo informację dla użytkownika. Pessimistic locking zakłada, że konflikt jest prawdopodobny, więc rekord jest blokowany wcześniej, na przykład przez `SELECT FOR UPDATE`. To może być bezpieczniejsze przy krytycznych operacjach, ale zmniejsza przepustowość i może prowadzić do deadlocków, jeśli transakcje są długie albo blokady są zakładane w różnej kolejności.

Częstym problemem aplikacji korzystających z ORM-ów jest N+1. Polega on na tym, że aplikacja wykonuje jedno zapytanie po listę encji, a następnie dla każdej encji osobne zapytanie po dane zależne. Dla kilku rekordów problem może być niewidoczny, ale dla setek lub tysięcy rekordów liczba zapytań rośnie liniowo i aplikacja zaczyna spędzać czas na round-tripach do bazy. N+1 można rozpoznać w logach SQL, metrykach albo profilerze. Naprawia się go najczęściej przez `fetch join`, entity graph, batch fetching albo zmianę sposobu pobierania danych.

Nie każda naprawa N+1 jest automatycznie dobra. Jeden wielki `fetch join` może doprowadzić do ogromnej liczby zduplikowanych wierszy, problemów z paginacją albo eksplozji danych przy wielu relacjach `OneToMany`. Czasem lepsze jest batch fetching, czasem osobne zapytanie agregujące dane, a czasem pozostawienie prostszego rozwiązania, jeśli liczba rekordów jest mała i nie ma realnego problemu wydajnościowego. Wydajność w SQL polega na świadomym wyborze kompromisu, a nie na stosowaniu jednej techniki do wszystkiego.

Dobrze opanowany SQL oznacza, że potrafisz nie tylko napisać zapytanie, ale również przewidzieć jego koszt, dobrać indeks, zinterpretować plan wykonania, dobrać poziom izolacji, rozpoznać problemy współbieżności i uzasadnić decyzję projektową. SQL nie jest wyłącznie językiem zapytań. W backendzie jest jednym z najważniejszych narzędzi do utrzymywania spójności, bezpieczeństwa i przewidywalnej wydajności systemu.
